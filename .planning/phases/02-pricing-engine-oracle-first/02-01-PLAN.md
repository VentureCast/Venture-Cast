   - buyCostCents(s0, Δ) = P0_cents*Δ + halfUpDiv(k_num*(s1^2 - s0^2), 2*k_den), s1 = s0+Δ
   - halfUpDiv(a, b) = floor(a/b) + (a%b*2 >= b ? 1 : 0)
   - Oracle (P0_cents=100, k_num=1, k_den=10, s0=1000, Δ=100) → gross 20500c, avg 205c, end 210c.
   - TEST-01 (oracle) is the FIRST test written (RED→GREEN).
3. ROUNDING RULE (LOCKED — solvency): every rounding favors the reserve/platform, never the user.
   Buy cost rounds UP (halfUpDiv; residual → reserve). Sell payout uses the same symmetric integral.
   Fees/spreads on a sell are deducted and on a buy are added so the user never gains a sub-cent.
   Worked non-zero-remainder example MUST be pinned: buyCostCents(1000, 4) → 801c (true float 800.80c).
4. Fee (feeBps, default 100) and spread (spreadBps, default 50) are TWO distinct integer-cent outputs:
   spread is the reserve's cut, fee is platform_fees' cut. The pricing functions RETURN integers;
   they do NOT touch any ledger (that is Phase 4). Fee math: floor(grossCents * bps / 10000).
5. Add a Number.isSafeInteger overflow guard on the s^2 term inside buyCostCents/sellPayoutCents.
   Throw PricingError if k_num*(s1*s1 - s0*s0) is not a safe integer (research: safe to ~4.5e13).
6. config.js holds per-tier { spreadBps, feeBps, reserveFloorCents } placeholders for tiers '1','2','3'.

OUT of scope (later phases): DB, risk checks, execution orchestrator, API, quote expiry.
